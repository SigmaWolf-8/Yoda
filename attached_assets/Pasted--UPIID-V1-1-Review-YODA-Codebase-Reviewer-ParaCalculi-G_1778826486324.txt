
Cargo reads the workspace manifest first. If any member path is absent,

freezes the build.

### BLOCKER 2 -- Private git dependency: ternary-math

```toml
ternary-math = { git = "https://github.com/SigmaWolf-8/Ternary" }
```

Pinned in Cargo.lock to commit `43678a714036c5a9136c572fbe07c5a9d5a6f283`.
The Ternary repo is private. On Replit, if GitHub auth is not configured
(SSH key or PAT in git config), cargo hangs waiting for credentials.
If the repo is being restructured and that commit was force-pushed away,
cargo fails even with valid auth.

**Impact:** `ternary-math` blocks ALL 6 remaining crates:

```
ternary-math
  └─ yoda-plenumnet-bridge
       ├─ yoda-inference-router
       │    ├─ yoda-orchestrator
       │    │    ├─ yoda-task-bible
       │    │    └─ yoda-api
       │    ├─ yoda-knowledge-base
       │    │    └─ yoda-api
       │    └─ yoda-api
       ├─ yoda-orchestrator
       ├─ yoda-knowledge-base
       ├─ yoda-task-bible
       └─ yoda-api
```

Every crate depends on `yoda-plenumnet-bridge` (even inference-router).
A single private git dep is a single point of failure for the entire workspace.

**Fix options:**
- Configure GitHub PAT on Replit: `git config --global url."https://<PAT>@github.com/".insteadOf "https://github.com/"`
- Vendor ternary-math into the workspace as a path dependency
- Add `.cargo/config.toml` with `[patch]` pointing to local checkout

### yoda-plenumnet-bridge (1,138 lines | 6 files)

The crypto bridge. Clean code. Five modules wrapping five ternary-math
primitives (TIS-27, TL-DSA, Phase Encryption, TLSponge-385, TL-KEM).

**Findings:**
- Structurally sound. Good test coverage (7+ tests per module).
- `kem.rs` calls `crate::encryption::derive_key_from_kem` -- verified, exists.
- All ternary-math imports assume specific API surface:
  `tl_dsa::{self, TlDsaVariant}`, `tlsponge385::hash_bulk_tis`,
  `phase_encryption::{encrypt, decrypt, ...}`, `tl_kem::{keygen, encapsulate, ...}`.
  If ternary-math's API changed during the restructure, these all break.
- `ternary-math` itself depends on `wasm-bindgen`. This drags in the
  entire WASM compilation toolchain on native builds unless gated behind
  `#[cfg(target_arch = "wasm32")]`. Adds significant build time.

**Risk:** Medium. Code is correct IF ternary-math API is stable.

## STRUCTURAL FINDINGS

### Dependency fan-out from ternary-math

Every crate in the workspace depends on yoda-plenumnet-bridge, which
depends on ternary-math. This is a single point of failure:

- If ternary-math can't be fetched: 0/7 crates compile
- If ternary-math API changes: yoda-plenumnet-bridge breaks, then 0/7
- The crypto functions (TIS-27 hash, TL-DSA sign) are used for integrity
  and audit. They're called at runtime, not build-time. The compile-time
  coupling could be loosened with a trait abstraction if needed.

### wasm-bindgen in ternary-math

Cargo.lock shows ternary-math depends on wasm-bindgen. On a native Linux
build (Replit runs Linux), this compiles the entire wasm-bindgen toolchain
even if no WASM target is used. This adds 2-5 minutes to a clean build
and is likely contributing to the "frozen" perception.

### No .cargo/config.toml

The workspace Cargo.toml comments mention creating one for local
development, but none exists. On Replit, without a config override,
cargo must fetch from GitHub every time.
