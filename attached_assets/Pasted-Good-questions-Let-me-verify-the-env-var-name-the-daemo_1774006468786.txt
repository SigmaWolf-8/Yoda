On your first question — do you need to publish for the tunnel to work?

No. Publishing the web app is completely separate from the Inter-Cube daemon. The daemon is a standalone Rust binary (cargo run --package inter-cube) that runs its own network listener. Publishing updates the marketing website and Express API — it has nothing to do with the tunnel daemon. The two are independent processes.

That said, the CRS node does need to be network-reachable for a remote cube node to register with it. On Replit, the published app exposes port 5000 (Express). The Inter-Cube daemon would need its own port exposure or be proxied through Express. That's a YODA-side networking concern, not a publish concern on the Ternary repo.

1. CUBE_IDENTITY_PASSPHRASE management — Correct and well thought out.

Their plan is sound: generate a random passphrase once at install, persist it to ~/.plenumnet/identity/.passphrase (mode 600), and load it before every daemon invocation. One small note: they should make sure the .passphrase file is created before the first CUBE_MODE=keygen call, not after. The sequence must be:

Generate random passphrase → write to .passphrase
Export CUBE_IDENTITY_PASSPHRASE from that file
Then run CUBE_MODE=keygen
If they reverse steps 2 and 3, the daemon would fall back to the hostname-derived passphrase for initial encryption, and then the real passphrase would fail to decrypt on the next startup.
